## 1. Foundations of LLMs & GenAI — "Attention Is All You Need"

The 2017 Google paper *"Attention Is All You Need"* introduced the **Transformer** architecture, which replaced recurrent networks (RNNs/LSTMs) as the backbone for language models. The core insight: instead of processing tokens sequentially one-by-one, the model attends to **all tokens simultaneously** and learns which ones are relevant to each other.

This is the paper that started the modern AI era — GPT, Claude, Gemini are all based on this architecture.

> **Key takeaway:** The transformer's self-attention mechanism lets the model understand context across the entire input at once, not just the previous few words.

## 2. Tokens, Context Windows, and Cost

**Tokens** are the units an LLM reads and writes — roughly 1 token ≈ ¾ of an English word. "ChatGPT" = 1 token; "Hello world" = 2 tokens. Non-English languages and code often use more tokens per word.

```
"The quick brown fox" → ["The", " quick", " brown", " fox"] → 4 tokens
"Supercalifragilistic" → split into ~5-6 subword tokens
```

**Context window** = the maximum tokens the model can "see" at once (input + output combined). If you exceed it, the oldest content gets cut off.

| Model | Context Window |
|-------|---------------|
| GPT-4o | 128K tokens |
| Claude 3.5 Sonnet | 200K tokens |
| Gemini 1.5 Pro | 1M tokens |

**Cost** is charged per token (input and output priced separately):

| Model | Input (per 1M tokens) | Output (per 1M tokens) |
|-------|----------------------|------------------------|
| GPT-4o | $2.50 | $10.00 |
| Claude 3.5 Sonnet | $3.00 | $15.00 |
| Claude 3.5 Haiku | $0.80 | $4.00 |
| Gemini 1.5 Flash | $0.075 | $0.30 |

> **Engineering implication:** Longer prompts = higher cost. Context compression, caching, and model tiering (use cheap model for simple tasks) are key cost-control levers.

## 3. Exploring LLM APIs — Anthropic, OpenAI, Gemini

All three providers expose a similar REST API pattern: send a list of messages, receive a completion.

```python
# OpenAI
from openai import OpenAI
client = OpenAI(api_key="sk-...")
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What is attention?"}],
)
print(response.choices[0].message.content)

# Anthropic
import anthropic
client = anthropic.Anthropic(api_key="sk-ant-...")
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "What is attention?"}],
)
print(response.content[0].text)

# Google Gemini
import google.generativeai as genai
genai.configure(api_key="AIza...")
model = genai.GenerativeModel("gemini-1.5-pro")
response = model.generate_content("What is attention?")
print(response.text)
```

## 5. Challenges and Limitations of LLMs

| Challenge | Description |
|-----------|-------------|
| **Hallucination** | Model confidently generates false facts — e.g. makes up a citation, wrong ISIN code |
| **Knowledge cutoff** | Model has no knowledge of events after its training data cutoff date |
| **Context window limits** | Very long documents must be chunked; model can't "read" a 1,000-page PDF in one shot |
| **No persistent memory** | Each API call is stateless — you must send conversation history every time |
| **Prompt injection** | Malicious text in user input or retrieved docs can override system instructions |
| **Inconsistency** | Same prompt → different answers; temperature and sampling introduce non-determinism |
| **Cost at scale** | 1M users × 500 tokens/query = $1,250/day on GPT-4o — cost explodes fast |
| **Bias and toxicity** | Trained on internet data; reflects societal biases |
| **No real reasoning** | Statistical next-token prediction — not true logical deduction (though reasoning models improve this) |

## 7. Deep Neural Network Based on Transformer Architecture

A Transformer is a stack of layers, each containing two main components:

```
Input tokens
    ↓
[Token Embeddings + Positional Encoding]
    ↓
┌─────────────────────────────────┐
│  Transformer Block × N          │
│  ┌───────────────────────────┐  │
│  │  Multi-Head Self-Attention │  │  ← "which tokens matter to each other?"
│  └───────────────────────────┘  │
│           ↓ (+ residual)        │
│  ┌───────────────────────────┐  │
│  │  Feed-Forward Network     │  │  ← "process each token's representation"
│  └───────────────────────────┘  │
│           ↓ (+ residual)        │
└─────────────────────────────────┘
    ↓
[Language Model Head]
    ↓
Probability over next token (vocab size ~50K–100K)
```

**Self-Attention** computes three vectors per token — **Q** (query), **K** (key), **V** (value) — and uses dot products to score how much each token should attend to every other token. Multi-head attention runs this in parallel with different learned projections, capturing different kinds of relationships.

## 8. Pretraining Dataset — FineWeb on Hugging Face

Before a model can be trained, you need **massive amounts of text**. The quality and diversity of this data directly determines the model's capabilities.

**FineWeb** (by Hugging Face) is a public, cleaned, high-quality web text dataset — 15 trillion tokens from CommonCrawl. The HuggingFace team published a detailed blog post on how it was built: [huggingface.co/spaces/HuggingFaceFW/blogpost-fineweb-v1](https://huggingface.co/spaces/HuggingFaceFW/blogpost-fineweb-v1)

Key dataset types used in LLM pretraining:

| Books | Books3, Project Gutenberg |
| Code | GitHub (The Stack) |
| Academic | ArXiv, PubMed |
| Conversational | Reddit (Pushshift), Wikipedia |

## 9. Processing and Filtering Before Training

Raw web crawl data is noisy — spam, duplicates, machine-generated junk, personal data. It must be heavily filtered before training.

**FineWeb's pipeline steps:**

1. **URL filtering** — remove adult content, spam domains, known low-quality sites
2. **Language filtering** — keep only English (or target language) using fastText
3. **Quality heuristics** — minimum word count, punctuation ratios, stop-word presence
4. **PII removal** — strip emails, phone numbers, credit card numbers
5. **Deduplication** — MinHash LSH to remove near-duplicate documents
6. **Safety filtering** — remove hate speech, CSAM

> Without this, models learn from garbage-in → garbage-out. Quality filtering is one of the most impactful improvements in modern LLMs.

## 10. CommonCrawl — Indexing the Web

**CommonCrawl** is a non-profit that continuously crawls the entire public web and releases the raw data for free. It's the foundation for most LLM training datasets.

- Crawls ~3 billion web pages per crawl cycle (roughly monthly)
- Stores raw HTML (WARC files), extracted text (WET files), and metadata (WAT files)
- Available for free on AWS S3 (`s3://commoncrawl/`)
- ~500TB per snapshot; multi-petabyte total archive

Most labs (OpenAI, Meta, Mistral, HuggingFace) start here and then apply their own filtering pipelines on top.

## 11. The Whole FineWeb Pipeline

```
CommonCrawl WARC files
    ↓
Text extraction (trafilatura / resiliparse)
    ↓
Language detection (fastText) → keep English
    ↓
Quality filters (heuristics: length, character ratios, etc.)
    ↓
Deduplication (MinHash at document level + line level)
    ↓
PII scrubbing (regex + NER models)
    ↓
Safety classifier (remove harmful content)
    ↓
FineWeb dataset (15T tokens, high quality)
    ↓
FineWeb-Edu subset (educational content scored by LLM classifier)
```

The **FineWeb-Edu** subset is particularly important — an LLM was used to score each document for educational quality, then only top-scoring documents were kept. This "LLM-as-filter" pattern is increasingly common.

## 12. Stages in Training an LLM

### Stage 1: Pretraining Dataset
Build or acquire a large, clean dataset (as above — CommonCrawl → FineWeb).

### Stage 2: Tokenization — Byte Pair Encoding (BPE)

Raw text can't be fed into a neural network directly — it must be converted to integers. **BPE** starts with individual bytes/characters and iteratively merges the most frequent adjacent pairs into new tokens.

```
"low" "lower" "newest" "widest"
→ initial vocab: {l, o, w, e, r, n, s, t, i, d}
→ merge most frequent pair: (e, r) → "er"
→ merge next: (l, o) → "lo"
→ final tokens: "low", "lo", "w", "er", "new", "est", ...
```

This creates a vocabulary of ~50K–100K tokens that efficiently represents any text, including code, math, and multiple languages.

## 13. Training the Neural Network

**The core loop (pretraining):**

1. Take a sequence of tokens: `["The", "cat", "sat", "on", "the"]`
2. Model predicts the next token: `"mat"` (with probability distribution over all ~50K vocab tokens)
3. Compare prediction to actual next token → compute **cross-entropy loss**
4. **Backpropagation**: compute gradient of loss w.r.t. every parameter in the network
5. **Gradient descent** (via Adam optimizer): nudge each weight slightly in the direction that reduces loss
6. Repeat billions of times across trillions of tokens

**Optimization = adjusting next-word prediction accuracy** by updating weights to make correct predictions more probable.

**Hardware:**
- **GPUs** (NVIDIA A100/H100): matrix multiplication accelerators; most common for training and inference
- **TPUs** (Google's Tensor Processing Units): custom ASICs optimised for TensorFlow/JAX; used by Google internally

**Inference** = running a trained model (no gradient updates, forward pass only). Much cheaper than training — but at scale (millions of requests), still significant.

## 14. Post-Training — Fine-Tuning (Instruction Tuning)

A pretrained model only predicts next tokens — it doesn't "follow instructions". Post-training shapes the model's behaviour.

**Supervised Fine-Tuning (SFT):**
- Create a dataset of (instruction, ideal response) pairs — e.g. `{"prompt": "Summarise this article", "response": "The article discusses..."}`
- Fine-tune the pretrained model on this dataset with the same next-token loss
- Model learns to respond helpfully in an instruction-following format
- This is how a raw LLM becomes a "chat" model

**Reinforcement Learning from Human Feedback (RLHF):**
- Goes further than SFT — teaches the model what *quality* means beyond just format
- See detailed section below ↓

## 15. Reinforcement Learning with Human Feedback (RLHF)

RLHF is the technique that turned raw language models into genuinely helpful, aligned assistants. It's why ChatGPT feels helpful rather than just statistically plausible.

### The Problem RLHF Solves
After SFT, the model follows instructions but has no concept of *better vs worse*. It might give a technically correct but verbose, misleading, or harmful answer. You need human judgement baked in — but you can't have a human review every single output at inference time.

### How It Works — Three Phases

**Phase 1: Supervised Fine-Tuning (SFT)**
- Start with a pretrained model
- Fine-tune on high-quality human-written instruction + response pairs
- Result: a baseline model that can follow instructions

**Phase 2: Reward Model Training**
- Collect **comparison data**: for the same prompt, generate 2–4 different responses
- Show these to human labellers and ask: *"Which response is better, and why?"*
- Train a separate **Reward Model (RM)** on these comparisons — a classifier that outputs a scalar score for any (prompt, response) pair
- The RM learns the pattern of what humans consider good: helpful, harmless, honest

**Phase 3: RL Optimisation (PPO)**
- The SFT model is now treated as a **policy** (in RL terms, the "agent")
- **Rollouts**: the policy generates responses for a large batch of prompts — these are the "actions" the agent takes in the "environment"
- Each generated response is scored by the Reward Model → **reward signal**
- **PPO (Proximal Policy Optimisation)** updates the policy to generate responses that score higher on the RM
- A **KL divergence penalty** is added to prevent the model from drifting too far from the SFT model (avoids reward hacking — gaming the RM with nonsensical high-scoring outputs)

```
Prompt → Policy (LLM) → Response (Rollout)
                                ↓
                        Reward Model → Score (e.g. 7.2 / 10)
                                ↓
                    PPO Update → nudge policy toward higher-reward responses
```

### What Grounds Human Feedback Is Taken On

Human labellers at OpenAI/Anthropic are given detailed rubrics scoring responses on:

| Dimension | Description |
|-----------|-------------|
| **Helpfulness** | Does it fully address what the user asked? Is it complete and actionable? |
| **Harmlessness** | Does it avoid generating harmful, dangerous, or toxic content? |
| **Honesty** | Does it acknowledge uncertainty? Does it avoid fabricating facts? |
| **Instruction following** | Does it match the requested format, length, tone? |
| **Coherence** | Is it logically consistent and well-structured? |

### Key Terms

- **Rollout**: a single generated response from the policy model for a given prompt — the "episode" in RL terminology
- **Human feedback**: pairwise comparison (A vs B) rather than absolute ratings — more reliable and consistent
- **Reward model**: a neural network trained on the comparisons to approximate human preference as a scalar score
- **PPO (Proximal Policy Optimisation)**: the RL algorithm used to update the LLM policy; "proximal" means it takes small, controlled steps to avoid catastrophic forgetting

## 16. RLHF vs RL — What It Is, Pros and Cons

### Classical RL (Reinforcement Learning)

An agent interacts with an **environment**, takes **actions**, receives **rewards** from the environment, and learns to maximise cumulative reward. The reward function is defined by the environment (e.g. score in a game, battery level in a robot).

```
Agent (policy) → Action → Environment → Reward + New State → Agent (update policy)
```

Used in: game playing (AlphaGo, Atari), robotics, recommendation systems.

### RLHF (Reinforcement Learning from Human Feedback)

The core difference: **there is no ground-truth environment reward**. For open-ended language tasks ("write a good poem"), there's no objective score. RLHF replaces the environment reward with a **learned reward model** trained on human preferences.

```
LLM (policy) → Generated text → Reward Model (proxy for human preference) → Scalar reward → PPO update
```

### Comparison Table

| Aspect | Classical RL | RLHF |
|--------|-------------|------|
| **Reward source** | Environment (objective, automatic) | Human labellers → trained reward model |
| **Reward type** | Sparse or dense, well-defined | Subjective, noisy, preference-based |
| **Action space** | Discrete (game moves) or continuous (robot joints) | Discrete tokens from vocab (50K+ options) |
| **Episode length** | Varies (seconds to hours) | One conversation turn (seconds) |
| **Scalability** | Scales automatically | Bottlenecked by human labelling cost |
| **Reward hacking risk** | High (can exploit game glitches) | High (can exploit reward model blind spots) |

### RLHF Pros
- Teaches nuanced human preferences that are hard to specify as rules
- Dramatically improves helpfulness, safety, and tone vs SFT alone
- Can be iterated as preferences evolve (re-collect data, retrain RM)
- Works even when "correct" output is subjective

### RLHF Cons
- Expensive: requires thousands of hours of human annotation
- Reward model is imperfect — the LLM can "hack" it (generate text that fools the RM but isn't actually good)
- Human labellers disagree — preferences are noisy and culturally variable
- Can suppress model capability if harmlessness reward is too strong (over-refusal)
- PPO is notoriously unstable and hard to tune at LLM scale

### Modern Alternatives to RLHF

| Method | Description |
|--------|-------------|
| **DPO (Direct Preference Optimisation)** | Skips the reward model entirely; directly fine-tunes on preference pairs — simpler, more stable |
| **GRPO (Group Relative Policy Optimisation)** | Used by DeepSeek-R1; no critic network, uses group-level relative rewards — very efficient |
| **Constitutional AI (Anthropic)** | Uses the model itself to critique and revise its outputs against a written "constitution" — reduces human annotation |
