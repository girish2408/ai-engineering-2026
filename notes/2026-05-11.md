## 1. Chat API Advanced Parameters

### `top_p` — Nucleus Sampling

An alternative to temperature for controlling randomness. Instead of scaling all token probabilities (temperature), `top_p` picks only from the **top tokens whose cumulative probability adds up to `p`**, and samples from that subset.

```python
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Describe the stock market"}],
    top_p=0.9,   # consider only the tokens that make up the top 90% probability mass
)
```

| `top_p` value | Effect |
|---------------|--------|
| `1.0` (default) | Consider all tokens — no filtering |
| `0.9` | Consider only high-probability tokens — slight focus |
| `0.1` | Very narrow — almost always picks the single most likely token |

> **Rule:** Use either `temperature` or `top_p` — not both at the same time. OpenAI recommends altering one and leaving the other at its default.

---

### `max_tokens` — Limit Output Length

Caps how many tokens the model can generate in its response. Useful for controlling cost and ensuring concise answers.

```python
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Summarise this article in one sentence: ..."}],
    max_tokens=100,   # stop after 100 output tokens even if not finished
)
```

> If the model hits `max_tokens` mid-sentence, the response is cut off. Set it generously for open-ended tasks and tightly for structured outputs.

---

### `n` — Multiple Answers in One Request

By default the API returns one completion (`n=1`). Setting `n > 1` returns multiple independent responses for the same prompt — useful for sampling diversity or picking the best answer.

```python
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Suggest a name for an AI wealth advisor product"}],
    n=3,         # return 3 different completions
    temperature=1.0,
)

# Access all choices
for i, choice in enumerate(response.choices):
    print(f"Option {i+1}: {choice.message.content}")

# Output:
# Option 1: WealthMind
# Option 2: Clarity AI
# Option 3: PortfolioIQ
```

> Each choice is a fully independent generation. Cost scales with `n` — you pay for all tokens across all choices.

---

### `stop` — Stop Sequences

Tell the model to stop generating as soon as it produces a specific string. The stop token itself is **not** included in the output. Can be a single string or a list of up to 4 strings.

```python
# Stop at a newline — good for single-line outputs
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "List one stock ticker:"}],
    stop="\n",
)

# Stop at multiple sequences
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Write a SQL query:"}],
    stop=["\n", ";"],   # stop at first newline OR semicolon
)
```

Common uses: stop at `"Human:"` or `"User:"` when building dialogue systems, stop at `"```"` after a code block, stop at `"\n\n"` for single-paragraph answers.

---

### `frequency_penalty` — Reduce Repetition of Already-Used Words

Range: **-2.0 to 2.0** (default `0`).

Penalises tokens **proportional to how many times they have already appeared** in the response. High values push the model to use new vocabulary; negative values encourage repeating words.

```python
# High frequency penalty — forces varied vocabulary
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Write a product description for a robo-advisor"}],
    frequency_penalty=1.5,   # strongly avoids reusing words
)

# Negative — model may deliberately echo/repeat for emphasis
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Write a chant for a trading floor"}],
    frequency_penalty=-1.0,
)
```

| Value | Effect |
|-------|--------|
| `2.0` | Aggressively avoids any repeated word |
| `0.5–1.0` | Mild variety push — good for creative writing |
| `0` | No penalty (default) |
| `-1.0` to `-2.0` | Encourages repetition — good for rhythmic/chant-like text |

---

### `presence_penalty` — Encourage Covering New Topics

Range: **-2.0 to 2.0** (default `0`).

A flat penalty applied to **any token that has appeared at least once**, regardless of how many times. This nudges the model to talk about new topics rather than rewording the same ideas.

```python
# High presence penalty — model explores new angles
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Tell me about investment risks"}],
    presence_penalty=1.5,   # once a topic is mentioned, model moves on to something new
)

# Negative — model stays focused on the same topic
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Explain compound interest in depth"}],
    presence_penalty=-1.0,  # stays on topic, goes deeper
)
```

**Frequency vs Presence — Key Difference:**
| Parameter | Penalty type | Effect |
|-----------|-------------|--------|
| `frequency_penalty` | Scales with count — more uses = bigger penalty | Reduces word repetition |
| `presence_penalty` | Flat — 1 use = same penalty as 10 uses | Encourages topic diversity |

## 2. Reasoning Models

### What Are They?

Reasoning models (e.g. OpenAI's `o1`, `o3`, `o3-mini`, `o4-mini`) are LLMs trained with reinforcement learning to **think step-by-step before answering**. They spend time internally "reasoning" — generating a hidden chain of thought — before producing the final visible output.

This makes them dramatically better at:
- Multi-step maths and logic
- Code debugging with complex constraints
- Scientific and analytical problems
- Tasks requiring careful planning

### Reasoning Tokens (In Context Window)

The model uses a portion of its context window for **reasoning tokens** — internal scratchpad thoughts that the user never sees. These count toward token usage and cost but are hidden from the API response.

```
User prompt: "What is the optimal rebalancing strategy for..."
     ↓
[REASONING TOKENS — hidden]
<thinking>
  Let me break this down: first I need to consider...
  The Sharpe ratio would be... if we assume...
  Actually wait, there's a constraint I missed...
</thinking>
     ↓
[VISIBLE RESPONSE]
"The optimal strategy is..."
```

In OpenAI's API response, you can see reasoning token count:
```python
print(response.usage.completion_tokens_details.reasoning_tokens)  # e.g. 1840
print(response.usage.completion_tokens_details.accepted_prediction_tokens)
```

### Key Parameters for Reasoning Models

```python
response = client.chat.completions.create(
    model="o4-mini",
    messages=[{"role": "user", "content": "Debug this algorithm..."}],
    reasoning_effort="high",     # low | medium | high — how much thinking to do
    max_completion_tokens=8000,  # caps total tokens (reasoning + output combined)
)
```

| Parameter | Values | Effect |
|-----------|--------|--------|
| `reasoning_effort` | `"low"`, `"medium"`, `"high"` | Controls how many reasoning tokens are used. `high` = slower, smarter, costlier |
| `max_completion_tokens` | integer | Total cap on reasoning + output tokens. Replace `max_tokens` for reasoning models |

> Use `max_completion_tokens` (not `max_tokens`) with reasoning models — `max_tokens` only caps visible output, not the reasoning budget.

## 3. Best Practices for Prompting Reasoning Models

Reasoning models behave differently from standard chat models. They already think carefully internally, so heavy prompt engineering can actually hurt performance.

---

### Use Developer Messages Instead of System Messages

Reasoning models (`o1` and later) use a `"developer"` role instead of `"system"` for instructions. It's the same position (before user messages) but signals to the model that these are trusted developer-level constraints.

```python
# ❌ Standard pattern — less effective with reasoning models
messages = [
    {"role": "system", "content": "You are a financial advisor..."},
    {"role": "user",   "content": "Analyse this portfolio..."},
]

# ✅ Reasoning model pattern
messages = [
    {"role": "developer", "content": "You are a financial advisor. Always show your calculation steps. Return JSON."},
    {"role": "user",      "content": "Analyse this portfolio..."},
]
```

---

### Keep Prompts Simple and Direct

Reasoning models do their own internal decomposition. You don't need to break down the problem for them — just state the goal clearly.

```python
# ❌ Over-engineered prompt
content = """
Step 1: Read the financial data
Step 2: Calculate the Sharpe ratio
Step 3: Compare to benchmark
Step 4: Form a recommendation
Now do all of this for the following data: ...
"""

# ✅ Direct prompt — model figures out the steps itself
content = "Calculate the Sharpe ratio for the following portfolio returns and compare it to the S&P 500 benchmark. Data: ..."
```

---

### Avoid Chain-of-Thought Prompting

Phrases like "think step by step", "let's reason through this", or "explain your reasoning" are useful for standard models but **redundant or harmful for reasoning models** — the model already reasons internally, and asking it to also show reasoning in output wastes tokens.

```python
# ❌ Unnecessary with reasoning models
content = "Think step by step. First consider X, then Y, then Z. Let's reason carefully..."

# ✅ Just ask for the answer
content = "What is the maximum drawdown of this portfolio? Return a single percentage."
```

---

### Use Delimiters for Clarity

When your prompt contains mixed content (instructions + data + examples), use clear delimiters to separate them so the model doesn't confuse one for another.

```python
content = """
Analyse the fund performance below and flag any anomalies.

<fund_data>
Date, NAV, Return
2026-01-01, 10.50, 0.5%
2026-02-01, 10.20, -2.9%
2026-03-01, 15.80, 54.9%   # suspicious spike
</fund_data>

<instructions>
- Flag returns outside ±10% as anomalies
- Explain the likely cause for each anomaly
- Return a JSON list
</instructions>
"""
```

Good delimiter options: `<tags>`, triple backticks ` ``` `, `---`, `###`.

---

### Provide Specific Guidelines

Vague instructions lead to vague outputs. Be explicit about format, length, tone, and what to include or exclude.

```python
# ❌ Vague
content = "Summarise this earnings report."

# ✅ Specific
content = """
Summarise the earnings report below.
- Output format: JSON with keys: revenue, net_income, yoy_growth, key_risks (list of 3)
- Length: one sentence per field (except key_risks)
- Exclude forward-looking guidance — facts only
- If a figure is missing, use null

<report>
...
</report>
"""
```

---

### Be Clear About the End Goal

State what success looks like — don't leave the model guessing at what you actually need.

```python
# ❌ Unclear end goal
content = "Look at this investment strategy."

# ✅ Clear end goal
content = """
Evaluate the following algorithmic trading strategy.
End goal: determine whether it is suitable for a risk-averse retail investor in the UAE.
Return: a verdict (Suitable / Not Suitable), confidence (0–100%), and 3 specific reasons.

<strategy>
...
</strategy>
"""
```

## 4. Whisper — OpenAI Speech-to-Text

**Whisper** is OpenAI's automatic speech recognition (ASR) model. It transcribes or translates audio files to text. It supports 99+ languages and is highly accurate even with accents and background noise.

```python
from openai import OpenAI

client = OpenAI()

# Transcription — audio → text in original language
with open("meeting_recording.mp3", "rb") as audio_file:
    transcript = client.audio.transcriptions.create(
        model="whisper-1",
        file=audio_file,
        language="en",            # optional — auto-detected if omitted
        response_format="text",   # "text" | "json" | "srt" | "vtt" | "verbose_json"
        prompt="ENBD, NAV, ISIN", # optional — hint vocabulary for domain terms
    )

print(transcript)  # "The NAV of fund IE00B4L5Y983 increased by 2.3% this quarter..."

# Translation — audio in any language → English text
with open("arabic_call.mp3", "rb") as audio_file:
    translation = client.audio.translations.create(
        model="whisper-1",
        file=audio_file,
    )
print(translation.text)
```

**Response formats:**
| Format | Output | Use case |
|--------|--------|---------|
| `text` | Plain string | Simple transcription |
| `json` | `{"text": "..."}` | When you need a dict |
| `verbose_json` | Text + word-level timestamps + language | When you need timestamps |
| `srt` | Subtitle format with timestamps | Video captions |
| `vtt` | WebVTT subtitle format | Web video captions |

**Supported audio formats:** `mp3`, `mp4`, `mpeg`, `mpga`, `m4a`, `wav`, `webm`. Max file size: 25MB.

## 5. Text-to-Speech — OpenAI TTS

Convert text to natural-sounding audio. Useful for voice assistants, accessibility tools, and voice-first AI products.

```python
from pathlib import Path
from openai import OpenAI

client = OpenAI()

response = client.audio.speech.create(
    model="tts-1",           # "tts-1" (fast) | "tts-1-hd" (higher quality)
    voice="nova",            # see voices below
    input="Welcome to ENBD Wealth. Your portfolio is up 3.2% this month.",
    response_format="mp3",   # see formats below
    speed=1.0,               # 0.25 to 4.0 — playback speed multiplier
)

# Save to file
Path("welcome.mp3").write_bytes(response.content)

# Or stream directly
with client.audio.speech.with_streaming_response.create(
    model="tts-1",
    voice="alloy",
    input="Streaming audio to the client...",
) as response:
    response.stream_to_file("output.mp3")
```

### Voices

| Voice | Personality |
|-------|------------|
| `alloy` | Neutral, balanced |
| `echo` | Male, clear |
| `fable` | Warm, storytelling |
| `onyx` | Deep, authoritative |
| `nova` | Energetic, friendly |
| `shimmer` | Soft, calm |

### Output Formats

| Format | Use case |
|--------|---------|
| `mp3` | Default — universal playback, good compression |
| `opus` | Low latency streaming (best for real-time voice apps) |
| `aac` | iOS/Safari compatible |
| `flac` | Lossless — archival quality |
| `wav` | Raw, no compression — easy to process programmatically |
| `pcm` | Raw PCM samples — lowest latency, use for LiveKit/WebRTC pipelines |

> For real-time voice bots (Week 8 project), use `pcm` or `opus` — they have the lowest latency and work well with WebRTC pipelines.

## 6. DALL-E — Image Generation

DALL-E is OpenAI's image generation model. You describe an image in text; DALL-E generates it.

```python
from openai import OpenAI

client = OpenAI()

response = client.images.generate(
    model="dall-e-3",            # "dall-e-2" | "dall-e-3"
    prompt="A futuristic bank branch in Dubai with holographic displays showing live stock charts, photorealistic",
    style="vivid",               # "vivid" (dramatic) | "natural" (subtle, realistic)
    size="1792x1024",            # see sizes below
    quality="hd",                # "standard" | "hd" (dall-e-3 only)
    n=1,                         # number of images (dall-e-3: max 1; dall-e-2: max 10)
    response_format="url",       # "url" (expires in 1h) | "b64_json" (base64 string)
)

image_url = response.data[0].url
print(image_url)

# dall-e-3 also returns the revised prompt (model may expand your prompt)
print(response.data[0].revised_prompt)
```

### Model Comparison

| Feature | DALL-E 2 | DALL-E 3 |
|---------|---------|---------|
| Quality | Good | Significantly better |
| Prompt following | Moderate | Excellent |
| Max `n` | 10 | 1 |
| Editing / variation | Yes | No (generate only) |
| Styles | N/A | `vivid` / `natural` |

### Size Options

| Model | Available sizes |
|-------|----------------|
| DALL-E 2 | `256x256`, `512x512`, `1024x1024` |
| DALL-E 3 | `1024x1024`, `1024x1792`, `1792x1024` |

### Quality

| Value | Description |
|-------|-------------|
| `standard` | Faster, cheaper — good for prototyping |
| `hd` | Higher detail, better consistency — use for production assets |

## 7. DALL-E — Create Variation

Take an existing image and generate variations of it — similar style and composition but different details. Only available on **DALL-E 2**.

```python
from openai import OpenAI

client = OpenAI()

# Generate variations of an existing image
with open("logo.png", "rb") as image_file:
    response = client.images.create_variation(
        model="dall-e-2",
        image=image_file,       # must be a square PNG, max 4MB
        n=3,                    # how many variations to generate
        size="1024x1024",
        response_format="url",
    )

for img in response.data:
    print(img.url)
```

**Constraints:**
- Input image must be a **square PNG** (same width and height)
- Max file size: 4MB
- No text prompt — the variation is purely style/content variation of the source image
- Only DALL-E 2 supports this — DALL-E 3 does not

## 8. DALL-E — Editing Images (Inpainting)

Edit a specific region of an image using a **mask**. The mask defines which pixels to regenerate — white pixels in the mask = areas to edit, black pixels = keep original. Only available on **DALL-E 2**.

```python
from openai import OpenAI

client = OpenAI()

with open("office.png", "rb") as image_file, \
     open("mask.png",   "rb") as mask_file:

    response = client.images.edit(
        model="dall-e-2",
        image=image_file,   # original image — square PNG, max 4MB
        mask=mask_file,     # mask PNG — white = regenerate, black = keep
        prompt="A large window with a view of Dubai Marina at sunset",
        n=1,
        size="1024x1024",
        response_format="url",
    )

print(response.data[0].url)
```

**How the mask works:**

```
Original image:   [office interior with blank wall on right side]
Mask PNG:         [all black except white region where the wall is]
Prompt:           "A large window with a view of Dubai Marina at sunset"
Result:           [same office, but the wall is now a floor-to-ceiling window]
```

**Constraints:**
- Both `image` and `mask` must be **square PNGs**, same dimensions, max 4MB
- Mask must have an **alpha channel** (transparent areas = edit zone) OR use a white/black mask
- Only DALL-E 2 supports editing — DALL-E 3 does not
- The model uses both the prompt and the surrounding image context to make the edit look seamless

**Practical AI engineering use case:** Generate product mockups (e.g. swap a background behind a product photo), redact/replace sensitive content in images, or generate annotated diagram variations.

## 9. Prompt Engineering — Guidelines

Prompt engineering is the practice of designing inputs to LLMs that consistently produce accurate, useful, and safe outputs. Small changes in phrasing can have dramatic effects on output quality.

**Two meta-rules before anything else:**
1. **Use the latest model** — newer models follow instructions more faithfully, hallucinate less, and handle complex prompts better. Don't optimise prompts for old models.
2. **Write clear and specific instructions** — vague prompts produce vague outputs. More specificity is almost always better, even if the prompt gets longer.

---

### 9.1 Put Instructions at the Top, One Per Line, with Delimiters

Place your instructions at the very **beginning** of the prompt before any data or content. List each instruction on its own line. Use **delimiters** to visually and semantically separate instructions from content — so the model never confuses "what to do" with "what to do it on".

**Why it matters:** Models have primacy bias — the beginning of the prompt is weighted more heavily. Instructions buried after a wall of text are more likely to be ignored or overridden.

**Good delimiters:**
| Delimiter | Use case |
|-----------|---------|
| Triple backticks ` ``` ` | Code, raw data, file contents |
| Triple quotes `"""` | Long text passages, articles, reports |
| `<tags>` | Labelled sections (e.g. `<context>`, `<document>`, `<rules>`) |
| `---` | General section separator |
| `{ }` | Inline variable placeholders |

```python
prompt = """
You are a financial compliance officer.
Instructions:
- Identify any regulatory violations in the document below
- List each violation on a new line
- Format: [Severity: HIGH/MEDIUM/LOW] — [Description]
- If no violations are found, respond with "COMPLIANT"

Document:
\"\"\"
{document_text}
\"\"\"
"""
```

```python
# Another example using triple backticks
prompt = """
Summarise the following earnings call transcript.
- Focus only on forward guidance and risk factors
- Output as bullet points, max 5 bullets

Transcript:
```
{transcript_text}
```
"""
```

---

### 9.2 Be Specific, Descriptive, and Detailed

Vague prompts produce vague answers. Specify the **desired outcome**, the **context** (who is the audience, what domain), and the **length** of the expected output.

Think of it like briefing a contractor: the more detail you give upfront, the less back-and-forth you need.

```python
# ❌ Vague — model makes too many assumptions
prompt = "Write about mutual funds."

# ✅ Specific — model has everything it needs
prompt = """
Write a 200-word explanation of mutual funds for a first-time retail investor in the UAE.
- Assume no prior financial knowledge
- Use simple, jargon-free language
- End with one concrete example using AED amounts
- Tone: friendly and encouraging
"""
```

```python
# ❌ Vague output spec
prompt = "Analyse this portfolio."

# ✅ Specific output spec
prompt = """
Analyse the portfolio data below.
Return a JSON object with these exact keys:
  - total_value (float, AED)
  - asset_allocation (dict: asset_class → percentage)
  - top_3_holdings (list of dicts: name, weight, ytd_return)
  - risk_rating (one of: "Conservative", "Moderate", "Aggressive")
  - one_line_summary (string, max 30 words)

Portfolio data:
\"\"\"
{portfolio_data}
\"\"\"
"""
```

### 9.3 RTF Framework — Role, Task, Format

**RTF** is a simple template that covers the three things every good prompt needs:

| Letter | Meaning | What it does |
|--------|---------|-------------|
| **R** — Role | Who the model should be | Sets expertise, tone, perspective |
| **T** — Task | What the model should do | The actual job to perform |
| **F** — Format | How the output should look | Structure, length, style |

```python
# RTF applied to a wealth advisory use case
prompt = """
Role:
You are a senior CFA-qualified wealth advisor with 15 years of experience in UAE equity markets.

Task:
Review the client's current portfolio allocation below and identify the top 3 rebalancing actions
they should take to reduce concentration risk while maintaining their target return of 8% annually.

Format:
- Return exactly 3 bullet points
- Each bullet: Action → Rationale → Expected Impact
- Be specific (use fund names and percentages, not general advice)
- Max 60 words per bullet

Portfolio:
\"\"\"
{portfolio_data}
\"\"\"
"""
```

```python
# RTF for a code generation task
prompt = """
Role: You are a senior Python engineer specialising in async FastAPI applications.

Task: Write an async endpoint that accepts a list of fund ISINs, calls the OpenAI API
in parallel for each one to generate a 1-sentence description, and returns a dict
mapping ISIN → description.

Format:
- Return only the Python function, no explanation
- Use asyncio.gather for parallel calls
- Include type hints
- Handle HTTPException for API failures
"""
```

### 9.4 Zero-Shot and Few-Shot Prompting

**Zero-shot:** Ask the model to perform a task with no examples — just instructions. Works well for tasks the model has seen often in training.

```python
# Zero-shot — no examples provided
prompt = """
Classify the sentiment of the following financial news headline as:
Positive, Negative, or Neutral.

Headline: "UAE central bank raises interest rates by 25 basis points amid inflation concerns"
"""
# Model answers from training knowledge alone: "Negative"
```

**Few-shot:** Provide 2–5 examples of input → output pairs before your actual request. Teaches the model the exact pattern you want — especially useful for custom formats, domain-specific classifications, or unusual output structures.

```python
# Few-shot — show the pattern, then ask the real question
prompt = """
Classify financial news headlines as Positive, Negative, or Neutral.

Examples:
Headline: "ENBD reports record profit of AED 16 billion in 2025"
Classification: Positive

Headline: "Oil prices fall 8% amid oversupply fears"
Classification: Negative

Headline: "ADIB announces unchanged dividend policy for Q2"
Classification: Neutral

Headline: "FAB shares drop 4% following CEO resignation announcement"
Classification: Negative

Now classify this headline:
Headline: "Emirates NBD launches new digital wealth management platform"
Classification:
"""
# Model learns the exact format and applies it: "Positive"
```

**When to use each:**
| Approach | Use when |
|----------|---------|
| Zero-shot | Task is common and well-defined (translation, basic classification) |
| Few-shot | Custom output format, domain-specific labels, unusual patterns, when zero-shot is inconsistent |

---

### 9.5 Specify the Steps Required to Complete a Task

For complex multi-part tasks, explicitly number the steps the model should follow. This forces sequential reasoning and prevents the model from skipping steps or collapsing multiple steps into a shallow answer.

```python
prompt = """
You are analysing a client's financial situation. Complete the following steps in order:

Step 1: Calculate the client's current asset allocation as a percentage breakdown by asset class.
Step 2: Identify which asset class is overweight relative to the target allocation provided.
Step 3: Calculate how much (in AED) needs to be shifted to reach the target allocation.
Step 4: Suggest two specific funds to sell from and two to buy into, with reasoning.
Step 5: Summarise the recommended rebalancing in one sentence.

Show your work for each step clearly before moving to the next.

Client portfolio: {portfolio_data}
Target allocation: {target_allocation}
"""
```

```python
# Step-by-step for a data extraction task
prompt = """
Process the bank statement below in these steps:

Step 1: Extract all transactions with amounts over AED 10,000.
Step 2: Categorise each extracted transaction as: Salary, Investment, Transfer, or Other.
Step 3: Sum the total by category.
Step 4: Flag any category where the total exceeds AED 50,000 as "High Activity".
Step 5: Return the result as JSON.

Statement:
\"\"\"
{statement_text}
\"\"\"
"""
```

### 9.6 Avoid Hallucinations Using Guarding

LLMs hallucinate — they generate plausible-sounding but false information. For production AI applications (especially finance, medical, legal) you must actively guard against this.

**Guarding techniques:**

**1. Instruct the model to say "I don't know"**

```python
prompt = """
Answer the question below using only the provided context.
If the answer is not explicitly stated in the context, respond with:
"I don't have enough information to answer this question."
Do NOT make up or infer information that isn't in the context.

Context:
\"\"\"
{retrieved_context}
\"\"\"

Question: {user_question}
"""
```

**2. Ask the model to cite its sources**

```python
prompt = """
Answer the question using the documents below.
For every claim you make, cite the document number it came from using [Doc N] notation.
```

**5. Schema validation as a guardrail (Pydantic)**

Even after careful prompting, always validate structured outputs programmatically:

```python
from pydantic import BaseModel, validator
from openai import OpenAI
import json

class FundAnalysis(BaseModel):
    fund_name: str
    risk_rating: str
    expected_return: float
    recommendation: str

    @validator("risk_rating")
    def rating_must_be_valid(cls, v):
        allowed = {"Conservative", "Moderate", "Aggressive"}
        if v not in allowed:
            raise ValueError(f"risk_rating must be one of {allowed}")
        return v

client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "Return valid JSON only."},
        {"role": "user",   "content": f"Analyse this fund: {fund_data}"},
    ],
    response_format={"type": "json_object"},
)

raw = json.loads(response.choices[0].message.content)
analysis = FundAnalysis(**raw)   # raises ValidationError if model output is wrong shape
```

> **Golden rule for production:** Never trust raw LLM output. Always validate, constrain, and provide fallback behaviour.
