## 1. LLMs with Internet Access — Search, Interpreter, Deep Research

Out of the box, LLMs have a knowledge cutoff and no access to live data. Three patterns extend them:

### Web Search (Internet Search)
The model is given a **search tool** — it formulates a query, calls a search API (Bing, Google, Brave, Tavily), gets results back as context, then generates an answer grounded in fresh data.

```
User: "What's the latest Fed interest rate decision?"
    ↓
LLM decides to call: search("Federal Reserve interest rate May 2026")
    ↓
Search API returns top 5 results (title + snippet)
    ↓
LLM reads snippets → generates answer with sources cited
```

Providers: OpenAI (`gpt-4o` with web search tool), Perplexity, Gemini (grounding with Google Search), Claude (via Brave/Tavily MCP server).

### Code Interpreter
The model is given a **sandboxed Python environment**. It writes code, executes it, reads the output, and iterates. Lets the model do exact calculations, data analysis, chart generation, and file processing — things pure language generation can't do reliably.

```
User: "Plot the monthly revenue trend from this CSV"
    ↓
LLM writes: pd.read_csv(...); df.plot(...)
    ↓
Code runs in sandbox → chart returned as image
    ↓
LLM describes the trend
```

OpenAI's **Code Interpreter** (in ChatGPT) and **Assistants API** both use this pattern.

### Deep Research / Deep Search
A multi-step agentic loop that combines web search + reasoning + synthesis. Instead of one search → one answer, the model:

1. Decomposes the question into sub-questions
2. Runs multiple searches iteratively, following links
3. Evaluates source quality and contradictions
4. Synthesises a long-form, cited report

OpenAI's **Deep Research**, Gemini's **Deep Research**, and Perplexity's **Deep Research** all use this pattern. The underlying approach is a ReAct-style loop (Reason → Search → Read → Reason → ...) running for minutes rather than milliseconds.

## 2. FastAPI

FastAPI is a modern Python web framework for building APIs. It is built on top of **Starlette** (ASGI server) and **Pydantic** (data validation). It has become the default choice for AI/ML APIs because it's fast to write, fast at runtime, and integrates naturally with Python async patterns.

### Structure of a FastAPI App

```
project/
├── main.py          ← app entry point, route definitions
├── models.py        ← Pydantic request/response schemas
├── routers/
│   ├── chat.py      ← route group for /chat endpoints
│   └── files.py     ← route group for /files endpoints
└── requirements.txt
```

```python
# main.py — minimal FastAPI app
from fastapi import FastAPI

app = FastAPI(title="AI API", version="1.0")

@app.get("/")
def root():
    return {"status": "ok"}
```

Run with: `uvicorn main:app --reload`

---

### Plain Python Syntax

FastAPI uses standard Python functions and type hints — no decorators magic, no special base classes. If you know Python, you know most of FastAPI already.

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class ChatRequest(BaseModel):
    message: str
    temperature: float = 0.7

@app.post("/chat")
def chat(req: ChatRequest):
    return {"reply": f"You said: {req.message}"}
```

---

### Built-in Asynchronicity — ASGI

FastAPI is built on **ASGI** (Asynchronous Server Gateway Interface) — the async successor to WSGI. This means routes can be `async def`, allowing non-blocking I/O. Critical for AI apps that call external LLM APIs (which are slow network calls — you don't want to block the server thread while waiting).

```python
import httpx
from fastapi import FastAPI

app = FastAPI()

@app.get("/llm")
async def call_llm(prompt: str):
    async with httpx.AsyncClient() as client:
        # non-blocking — other requests can be served while waiting
        r = await client.post("https://api.openai.com/v1/...", ...)
    return r.json()
```

**WSGI vs ASGI:**
| | WSGI (Flask, Django) | ASGI (FastAPI, Starlette) |
|-|---------------------|--------------------------|
| I/O model | Synchronous (blocking) | Asynchronous (non-blocking) |
| Concurrent requests | One thread per request | Many requests per thread |
| Good for | Traditional web apps | API servers, AI backends, streaming |

### Integrated Type Hints and Data Validation with Pydantic

FastAPI uses **Pydantic models** to define request and response shapes. It automatically validates incoming JSON, coerces types, and returns a 422 error if validation fails — zero manual validation code.

```python
from pydantic import BaseModel, Field
from typing import Literal

class ChatRequest(BaseModel):
    message: str = Field(..., min_length=1, max_length=4000)
    temperature: float = Field(0.7, ge=0.0, le=2.0)
    model: Literal["gpt-4o", "gpt-4o-mini"] = "gpt-4o"

# FastAPI auto-validates this on every POST /chat request
# If temperature=5.0 is sent → 422 Unprocessable Entity returned automatically
```

Pydantic also generates the OpenAPI schema (used for Swagger UI) automatically from these models.

### Built-in Error Handling and JSON Errors

FastAPI returns structured JSON errors by default — no extra config needed.

```python
from fastapi import FastAPI, HTTPException

app = FastAPI()

@app.get("/fund/{isin}")
def get_fund(isin: str):
    fund = db.get(isin)
    if not fund:
        raise HTTPException(status_code=404, detail=f"Fund {isin} not found")
    return fund

# Response on 404:
# {"detail": "Fund IE00B4L5Y983 not found"}
```

You can also override the default exception handlers for custom error shapes:

```python
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse

@app.exception_handler(RequestValidationError)
async def validation_exception_handler(request, exc):
    return JSONResponse(status_code=422, content={"error": "Invalid input", "detail": str(exc)})
```

### Built-in Authentication

FastAPI has built-in support for common auth patterns via `fastapi.security`:

```python
from fastapi import FastAPI, Depends, HTTPException
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

app = FastAPI()
security = HTTPBearer()

def verify_token(credentials: HTTPAuthorizationCredentials = Depends(security)):
    if credentials.credentials != "my-secret-token":
        raise HTTPException(status_code=401, detail="Invalid token")
    return credentials.credentials

@app.get("/protected", dependencies=[Depends(verify_token)])
def protected_route():
    return {"data": "secret"}
```

The `Depends()` system is FastAPI's dependency injection — clean, composable, testable.

### Automatic Interactive Documentation — Swagger UI and ReDoc

FastAPI auto-generates two interactive docs from your code — no extra work needed:

| URL | Interface | Best for |
|-----|-----------|---------|
| `/docs` | Swagger UI | Testing endpoints interactively in browser |

## 3. fastapi-mcp

`fastapi-mcp` is a library that automatically exposes every FastAPI route as an **MCP (Model Context Protocol) tool** — so an AI agent (Claude, etc.) can discover and call your API endpoints as tools, without any extra configuration.

```bash
pip install fastapi-mcp
```

```python
from fastapi import FastAPI
from fastapi_mcp import FastApiMCP

app = FastAPI()

@app.get("/portfolio/{account_id}")
def get_portfolio(account_id: str):
    """Get portfolio holdings for a given account."""
    return {"account": account_id, "holdings": [...]}

# Attach MCP — exposes all routes as MCP tools
mcp = FastApiMCP(app)
mcp.mount()
# MCP server now available at /mcp
```

What this gives you:
- Claude (or any MCP-compatible client) connects to `/mcp` and discovers `get_portfolio` as a callable tool
- The tool description, parameters, and types are pulled directly from your FastAPI route definition and docstring
- Zero duplication — your API and your MCP tools are defined in one place

This is the pattern to use in Week 5 (MCP server project).

## 4. OpenAI Chat Completions API

The **Chat Completions** endpoint is the core of the OpenAI API. It takes a list of messages (a conversation) and returns the model's next reply.

```python
from openai import OpenAI

client = OpenAI(api_key="sk-...")

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "You are a helpful financial advisor."},
        {"role": "user",   "content": "What is a mutual fund?"},
    ],
)

print(response.choices[0].message.content)
print(response.usage)  # prompt_tokens, completion_tokens, total_tokens
```

**Message roles:**
| Role | Purpose |
|------|---------|
| `system` | Sets the model's persona, instructions, constraints — runs at the start |
| `user` | The human's turn — what the user sent |
| `assistant` | The model's previous replies — include for multi-turn conversations |

**Multi-turn conversation pattern:**

```python
messages = [{"role": "system", "content": "You are a wealth advisor."}]

# Turn 1
messages.append({"role": "user", "content": "What is an ETF?"})
reply = client.chat.completions.create(model="gpt-4o", messages=messages)
assistant_reply = reply.choices[0].message.content
messages.append({"role": "assistant", "content": assistant_reply})

# Turn 2 — model has context of turn 1
messages.append({"role": "user", "content": "How does it differ from a mutual fund?"})
reply2 = client.chat.completions.create(model="gpt-4o", messages=messages)
```

## 5. OpenAI Parameters — Temperature, Seed, Base64 Images, Streaming, System Prompt, User Prompt

### Temperature

Controls randomness in token sampling. Range: `0.0` to `2.0`.

```python
# Low temperature — deterministic, focused, factual
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "What is 2+2?"}],
    temperature=0.0,   # always "4"
)

# High temperature — creative, varied, sometimes surprising
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Write a poem about markets"}],
    temperature=1.4,   # more surprising word choices
)
```

| Temperature | Use case |
|-------------|---------|
| `0.0` | Factual Q&A, data extraction, classification |
| `0.3–0.7` | Balanced — general chat, summaries |
| `0.8–1.2` | Creative writing, brainstorming |
| `1.2–2.0` | Experimental — very random, often incoherent |

---

### Seed

Makes outputs **reproducible** — given the same seed and inputs, the model will return (approximately) the same output. Useful for testing and debugging.

```python
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Describe a trading strategy"}],
    temperature=0.9,
    seed=42,   # same seed → same (or very similar) output every time
)
# Check if the system fingerprint changed (model update could break reproducibility)
print(response.system_fingerprint)
```

### Base64 Images (Vision)

GPT-4o is multimodal — it can read images. You can send images either as a URL or as a **base64-encoded string** (useful when the image isn't publicly accessible).

```python
import base64

# Read and encode image
with open("chart.png", "rb") as f:
    image_data = base64.b64encode(f.read()).decode("utf-8")

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {
            "role": "user",
            "content": [
                {"type": "text", "text": "Describe what you see in this chart."},
                {
                    "type": "image_url",
                    "image_url": {
                        "url": f"data:image/png;base64,{image_data}",
                        "detail": "high",   # "low" = faster/cheaper, "high" = more accurate
                    },
                },
            ],
        }
    ],
)
```

### Streaming API Responses

By default, the API waits until the full response is generated, then returns it — the user sees nothing until it's done. **Streaming** sends tokens as they are generated, giving a real-time typewriter effect.

```python
# Without streaming — waits for full response
response = client.chat.completions.create(model="gpt-4o", messages=[...])
print(response.choices[0].message.content)  # all at once

# With streaming — print tokens as they arrive
stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Write a market summary"}],
    stream=True,
)

for chunk in stream:
    delta = chunk.choices[0].delta.content
    if delta:
        print(delta, end="", flush=True)  # prints token by token
```

**FastAPI streaming endpoint:**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from openai import OpenAI

app = FastAPI()
client = OpenAI()

async def stream_llm(prompt: str):
    stream = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}],
        stream=True,
    )
    for chunk in stream:
        delta = chunk.choices[0].delta.content
        if delta:
            yield delta

@app.get("/stream")
async def stream_endpoint(prompt: str):
    return StreamingResponse(stream_llm(prompt), media_type="text/plain")
```

### System Prompt and User Prompt

The **system prompt** is the most important prompt you write as an AI engineer — it defines the model's persona, rules, and constraints for the entire conversation.

```python
# System prompt — set once, shapes all responses
system_prompt = """You are a financial advisor specialising in UAE wealth products.

Rules:
- Only discuss mutual funds, ETFs, and portfolio allocation
- Always cite uncertainty when discussing future performance
- Never provide specific buy/sell recommendations
- Respond in formal English; keep answers under 300 words
- If asked about anything outside finance, politely decline"""

# User prompt — the actual question from the user
user_prompt = "Should I invest in a tech ETF right now?"

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": system_prompt},
        {"role": "user",   "content": user_prompt},
    ],
    temperature=0.3,
)
```

**Best practices for system prompts:**
- Be explicit about what the model should AND should not do
- Define the output format if you need structured responses
- Set tone, persona, and language style
- Include few-shot examples inside the system prompt for complex tasks
- Put the most important constraints at the top and bottom (primacy and recency bias)
